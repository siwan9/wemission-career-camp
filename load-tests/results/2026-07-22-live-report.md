# 2026-07-22 배포 서버 부하 테스트 결과

대상: `https://wemission-career.site`

부하 발생기: 개발 PC 한 대

## 홈 새로고침

각 VU는 독립 세션으로 `/home`을 두 번 요청했다. 모집이 이미 `OPEN` 상태였으므로 이번 결과는 새로고침 부하만 검증하며, `WAITING`에서 `OPEN`으로 바뀌는 스케줄러 지연은 검증하지 않는다.

| VU | 홈 요청 | 실패 | 오픈 구간 p95 | 결과 |
|---:|---:|---:|---:|:---|
| 10 | 20 | 0 | 39ms | 통과 |
| 100 | 200 | 0 | 213ms | 통과 |
| 500 | 1,000 | 0 | 892ms | 통과 |
| 1,000 | 2,000 | 0 | 1.62s | 가용성 통과, 1초 성능 기준 실패 |
| 2,000 | 4,000 | 0 | 4.73s | 가용성 통과, 성능 기준 실패 |
| 3,000 | 6,000 | 1,099 (18.31%) | 4.78s | 실패 |

3,000 VU 실패는 HTTP 5xx 응답이 아니라 `EOF`와 `connection reset by peer`였다. 같은 시각 Nginx 로그에서 `1024 worker_connections are not enough while connecting to upstream`이 반복되어, 직접 원인은 Nginx의 `worker_connections=1024` 제한으로 확인됐다. 성공 요청도 있었으며 테스트 종료 직후 단일 `/home` 요청은 HTTP 200, 약 46ms로 회복됐다.

2,000 VU부터 부하 발생기 자체의 TLS 연결 생성 지연도 p95 약 4.3초로 증가했다. 3,000 VU 재검증은 두 대 이상의 부하 발생기에서 분산 실행해야 서버와 단일 PC 한계를 분리할 수 있다.

## Nginx 조정 후 홈 재검증

Nginx 연결 제한과 컨테이너 `nofile` 설정을 조정한 뒤 같은 개발 PC에서 동일한 방식으로 재실행했다.

| VU | 홈 요청 | 실패 | 홈 응답 p95 | 부하 발생기 TLS 준비 p95 | 결과 |
|---:|---:|---:|---:|---:|:---|
| 2,000 | 4,000 | 0 | 1.52s | 4.04s | 가용성 통과, 1초 성능 기준 실패 |
| 3,000 | 6,000 | 0 | 1.28s | 6.19s | 가용성 통과, 1초 성능 기준 실패 |

3,000 VU에서 사전 점검과 관찰 요청까지 포함한 전체 6,002건도 HTTP 실패가 0건이었다. 수정 전 1,099건(18.31%)이었던 연결 실패가 사라졌으므로 Nginx 연결 제한 조정 효과가 확인됐다. 테스트 종료 직후 `/home`은 HTTP 200, 약 51ms로 정상 응답했다.

현재 모집이 이미 `OPEN` 상태이므로 스크립트의 가상 오픈 시각 관찰 기준은 유효한 스케줄러 측정값이 아니다. 단일 개발 PC의 TLS 준비 지연도 3,000 VU에서 p95 6.19초까지 증가했으므로, 실제 사용자 관점의 3,000명 지연시간은 분산 부하 발생기로 다시 측정해야 한다.

## 배포 전 조치

1. `www.wemission-career.site` 인증서에 `www` SAN이 없어 TLS 검증이 실패한다. 인증서와 루트 도메인 리다이렉트를 수정한다.
2. Nginx 설정 적용값이 `worker_connections=8192`, 컨테이너 `nofile=65535`인지 배포 재시작 후에도 유지되는지 최종 확인한다.
3. 서로 다른 부하 발생기 두 대 이상에서 1,500 VU씩 분산해 3,000 VU의 실제 지연시간을 재검증한다.
4. DB 서버에서 `verify-registration-race.sql`을 실행해 저장 카운터와 실제 확정 신청 수가 일치하는지 최종 확인한다.
5. 실제 스케줄러 전환은 새로운 `WAITING` 리허설 모집을 만들어 별도로 검증한다.
